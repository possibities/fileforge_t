# 毕业设计边界与论文讲述提纲

本文用于统一论文、PPT 和系统演示的说法。核心原则是:问题真实、边界清楚、流程闭合、结果可验证。

> 当前会话若为非可执行环境,本文不记录运行结果；涉及实验、演示和截图的内容需要在目标环境中实际验证后再写入论文或答辩材料。

## 1 项目定位

推荐论文中的系统定位:

```text
本文设计并实现了一个面向档案整理场景的档案分类与元数据管理原型系统。系统以扫描档案图像为输入,经过 OCR 文本识别、结构化字段解析、规则校正、件号生成和结果导出,辅助完成题名、责任者、年度、分类号、保管期限等关键元数据的生成,并提供 Web 后台完成上传、处理进度查看、结果复核、人工修正和记录追溯。
```

不要把系统表述为完整档案馆平台。更准确的边界是:

```text
档案整理辅助系统原型
```

而不是:

```text
生产级智慧档案馆系统
完整档案管理平台
全流程档案生命周期管理系统
```

## 2 现实问题

论文的现实问题可以这样写:

```text
在纸质档案数字化和归档整理过程中,工作人员需要逐份阅读扫描件并填写题名、责任者、年度、分类号、保管期限等元数据。该过程重复性强、耗时长,且字段填写容易受到文本质量、业务规则和人工经验差异影响。针对这一问题,本文将 OCR、字段解析和确定性规则校正结合,构建一个能够辅助生成档案元数据并支持人工复核的原型系统。
```

这里的重点不是“替代档案员”,而是:

- 减少重复录入。
- 提供初始元数据。
- 通过规则降低明显错误。
- 给人工复核提供可追溯依据。

## 3 系统边界

### 3.1 本文实现的内容

| 模块 | 是否实现 | 说明 |
| --- | --- | --- |
| 单位管理 | 是 | 支持基本组织范围和权限归属 |
| 用户登录与角色 | 是 | 支持平台管理员、单位管理员、单位操作员 |
| 整理项目管理 | 是 | 表示一次档案整理任务 |
| 扫描件上传 | 是 | 支持图片和 zip 上传 |
| 在线跑批 | 是 | Web 后台触发处理批次 |
| OCR 识别 | 是 | 基于 PaddleOCR |
| 字段解析 | 是 | 通过外部 vLLM OpenAI 兼容接口完成 |
| 规则校正 | 是 | 对分类号、保管期限、开放状态、题名等字段做确定性修正 |
| 件号/档号生成 | 是 | 按年度、分类号、保管期限分组生成 |
| 档案结果查询 | 是 | 支持批次和档案结果查看 |
| 人工审核 | 是 | 审核工作台,唯一编辑入口,开放 8 个字段 |
| 修订记录和审计 | 是 | 记录人工修正和关键操作 |
| JSON/CSV 导出 | 是 | 输出可交付结果 |

### 3.2 本文不实现的内容

| 不实现内容 | 原因 |
| --- | --- |
| 借阅管理 | 属于档案利用环节,不属于整理阶段核心问题 |
| 库房管理 | 需要实体库位、出入库和盘点流程,超出本系统边界 |
| 销毁鉴定 | 涉及审批和法规流程,不适合纳入当前原型 |
| 多级审批流 | 会显著增加流程复杂度,不是本文重点 |
| 生产级任务队列 | 当前使用 Web 后台任务满足演示和小规模验证,队列化可作为后续工作 |
| 全文检索系统 | 可作为扩展方向,不是当前元数据生成闭环的必要条件 |
| Excel 模板适配 | 当前提供 JSON/CSV,模板化 Excel 可作为后续扩展 |

## 4 论文主线

建议围绕下面这条主线展开:

```text
档案整理元数据录入效率低
-> 使用 OCR 获取扫描件文本
-> 使用字段解析生成初始元数据
-> 使用规则引擎修正常见业务错误
-> 使用序号生成器生成档号和件号
-> 使用 Web 后台完成上传、处理、复核、修正和导出
```

每章都应服务这条主线,不要展开过多无关功能。

## 5 章节建议

### 第 1 章 绪论

建议包含:

- 档案数字化和整理工作的背景。
- 元数据录入工作量大、规则性强、易出错的问题。
- 本文目标:构建一个档案整理辅助原型系统。
- 本文贡献:
  - 设计 OCR、字段解析、规则校正、件号生成的处理流程。
  - 实现支持上传、处理、复核、修正和导出的 Web 后台。
  - 在脱敏样例集上验证流程可行性。

避免写:

```text
完全替代人工
适用于所有档案类型
达到生产级可靠性
```

### 第 2 章 相关技术

建议介绍:

- OCR 技术及扫描档案文本识别问题。
- 结构化信息抽取方法。
- 规则引擎在业务字段校正中的作用。
- Web 后台和数据库持久化技术。

不要堆太多与系统无关的大模型背景。重点写“为什么这些技术适合本问题”。

### 第 3 章 需求分析与系统设计

建议包含:

- 角色分析:平台管理员、单位管理员、单位操作员。
- 功能需求:
  - 单位与用户管理。
  - 整理项目管理。
  - 上传与在线处理。
  - 档案结果查看。
  - 人工修正与追溯。
  - 结果导出。
- 非功能需求:
  - 小规模演示稳定。
  - 异常不阻断结果交付。
  - 数据可追溯。
  - 敏感数据不进入仓库。

### 第 4 章 核心算法与处理流程

建议按实际管线写:

```text
扫描图像 -> OCR 文本 -> 初始元数据 -> 规则校正元数据 -> 档号/件号 -> JSON/CSV
```

可拆成:

- OCR 文本重建。
- 元数据字段解析。
- 规则引擎设计。
- 保管期限和分类号校正。
- 开放状态判定。
- 件号和档号生成。
- 失败兜底与异常处理。

### 第 5 章 系统实现

建议按模块写:

- `core/classifier.py`: 单份档案处理编排。
- `infrastructure/ocr_client.py`: OCR 和版面文本重建。
- `infrastructure/llm_client.py`: OpenAI 兼容接口调用和 JSON 解析。
- `core/rules_engine.py`: 业务规则校正。
- `core/sequence_generator.py`: 件号和档号生成。
- `processors/batch_processor.py`: 批量处理与错误隔离。
- `web_admin/`: Web 后台。
- `infrastructure/db/`: 数据库存储与查询。

这里可以结合类图或模块图,但不要贴大段代码。

### 第 6 章 测试与结果分析

建议至少包含:

- 单元测试情况。
- Web 主要流程测试。
- 小样本演示结果。
- 字段抽取效果表。
- 规则校正前后对比。
- 失败案例分析。

示例表格:

| 字段 | 样本数 | 正确数 | 准确率 | 主要错误来源 |
| --- | --- | --- | --- | --- |
| 题名 | 待填 | 待填 | 待填 | OCR 误识别/标题不完整 |
| 责任者 | 待填 | 待填 | 待填 | 签发单位不明显 |
| 年度 | 待填 | 待填 | 待填 | 文件日期缺失 |
| 分类号 | 待填 | 待填 | 待填 | 新旧分类码混淆 |
| 保管期限 | 待填 | 待填 | 待填 | 语义规则不足 |

不要编造数字。没有足够样本时,用“案例验证”和“流程验证”表述。

### 第 7 章 总结与展望

总结:

- 完成了档案整理辅助处理闭环。
- 实现了 Web 上传、在线跑批、结果复核、人工修正和导出。
- 通过规则校正提高字段一致性。

展望:

- 引入独立任务队列。
- 扩大样本规模。
- 增加更多档案类型。
- 增加 Excel 模板导出。
- 增强图像质量检测和上传预检。
- 引入更细粒度的复核流程。

## 7 答辩表述建议

### 7.1 开场 30 秒

```text
本系统面向纸质档案数字化后的整理环节。传统方式需要人工逐份阅读扫描件并填写题名、责任者、年度、分类号、保管期限等元数据。我的系统将 OCR、字段解析和规则校正组合起来,生成初始档案元数据,并提供 Web 后台完成上传、处理、复核、修正和导出。
```

### 7.2 强调边界

```text
系统定位是档案整理辅助原型,重点解决扫描件到结构化元数据的生成和复核问题。借阅、库房、销毁鉴定、多级审批等完整档案馆业务不在本文范围内。
```

### 7.3 回答“为什么要有整理项目”

```text
整理项目表示一次具体档案整理任务。单位负责权限归属,整理项目负责业务范围。上传批次、处理批次和档案结果都归属于整理项目,这样便于查询、追溯和演示。
```

### 7.4 回答“为什么还需要人工修正”

```text
档案字段具有业务规则和语义判断,自动处理适合生成初始结果并减少重复录入,但不应取消人工复核。系统保留人工修正入口和修订记录,可以把自动处理和人工把关结合起来。
```

## 8 避免过度承诺

论文和答辩中不要使用以下表述:

- “完全自动归档”
- “无需人工参与”
- “适用于所有档案”
- “生产级高并发系统”
- “准确率达到行业领先”

可替换为:

- “辅助生成”
- “减少重复录入”
- “支持人工复核”
- “在脱敏样例集上验证”
- “具备后续扩展空间”
