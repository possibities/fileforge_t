# 毕业论文与答辩准备材料清单

本文用于整理本项目作为毕业论文时还需要补齐、冻结和展示的材料。当前项目的工程主体已经较完整，后续重点应放在实验数据、论文图表、演示材料和文档一致性上。

> 当前会话若为非可执行环境,本文只作为静态准备清单；实验结果、截图、录屏、测试和系统运行状态必须在目标环境中实际确认。

## 1. 功能范围冻结

建议将当前系统功能范围冻结为：

- 扫描图像目录批处理。
- PaddleOCR 图像文字识别与版面重建。
- vLLM OpenAI 兼容接口调用大模型抽取元数据。
- 规则引擎进行确定性校正。
- 件号和档号生成。
- JSON/CSV 导出。
- 可选 PostgreSQL 旁路写库。
- Web 后台上传、在线跑批、查询、筛选、人工修正和审计追踪。
- CLI 工具用于账号初始化、查询和强制重跑。

建议不要继续扩张新的大功能，例如全文检索、复杂角色管理页、多 OCR 引擎切换、生产级任务队列、上传任务暂停/取消/重试等。这些可以放入“未来工作”。

论文边界建议表述为：

- 论文对象：支持 Web 上传、异步在线跑批、人工复核与可追溯审计的档案智能著录系统。
- 核心技术贡献：OCR 版面重建、LLM 元数据抽取、确定性规则后处理、三快照可追溯复核。
- 工程闭环贡献：Web 上传、在线跑批、数据库持久化、权限与审计让系统可演示、可使用、可验证。
- 不应夸大：Web 上传和后台任务属于系统工程实现，不单独包装为算法创新。

## 2. 实验数据准备

需要准备一批人工标注档案作为评测集，建议规模：

| 规模 | 用途 |
| --- | --- |
| 30 份 | 最小可用实验，适合时间紧张时使用 |
| 50 份 | 推荐规模，能支撑基本统计 |
| 100 份 | 更有说服力，适合答辩展示 |

每份档案建议标注以下字段作为真值：

- 门类。
- 归档年度。
- 实体分类号。
- 实体分类名称。
- 保管期限。
- 责任者。
- 文件编号。
- 题名。
- 文件形成时间。
- 密级。
- 保密期限。
- 开放状态。
- 延期开放理由。
- 立档单位名称。

建议保存一份人工标注表，例如：

- `实验数据编号`
- `档案文件夹`
- `页数`
- `人工题名`
- `人工责任者`
- `人工实体分类号`
- `人工保管期限`
- `人工开放状态`
- `人工延期开放理由`
- `备注`

## 3. 核心实验指标

论文中至少应报告以下指标：

| 指标 | 说明 |
| --- | --- |
| 字段级准确率 | 每个字段预测正确的比例 |
| 档案级完全正确率 | 一份档案所有核心字段均正确的比例 |
| 分类号准确率 | 实体分类号是否正确 |
| 保管期限准确率 | 保管期限是否正确 |
| 题名准确率或可接受率 | 题名是否符合著录要求 |
| 开放状态准确率 | 开放/控制判断是否正确 |
| JSON 解析成功率 | LLM 输出能否被稳定解析 |
| 批处理成功率 | 成功处理档案数 / 总档案数 |
| 平均处理耗时 | 单页 OCR、单份档案、整批平均耗时 |
| 人工修正率 | 需要人工修改的档案比例 |

建议将字段分为两类统计：

- 客观字段：归档年度、分类号、保管期限、密级、开放状态。
- 半主观字段：题名、责任者、备注。

## 4. 消融实验

建议至少完成以下消融对比：

| 实验组 | 目的 |
| --- | --- |
| 仅 LLM | 作为基础对照 |
| LLM + 规则引擎 | 验证规则引擎提升 |
| 原图 OCR | 作为 OCR 预处理对照 |
| 预处理增强 OCR | 验证低质量扫描件增强效果 |
| 普通 prompt | 作为提示词对照 |
| 当前分片 prompt + 示例 | 验证提示词工程效果 |
| 无简报二次重写 | 验证简报题名特殊路径 |
| 有简报二次重写 | 验证题名规范化能力 |

推荐重点写两组：

1. `仅 LLM` vs `LLM + 规则引擎`。
2. `原图 OCR` vs `预处理增强 OCR`。

这两组最容易说明项目的技术贡献。

## 8. 论文贡献点表述

建议将贡献点表述为：

1. 设计并实现了一套面向扫描档案的 OCR-LLM-规则融合式智能著录流程，能够从多页扫描图像中自动生成结构化档案元数据。
2. 构建了面向档案分类业务的确定性规则引擎，对大模型抽取结果中的分类号、保管期限、开放状态和题名等字段进行校正，提高关键字段的可靠性。
3. 设计了可追溯的人机协同机制，将模型抽取、规则修正和人工修正结果分层保存，并通过 Web 后台支持上传跑批、人工校核、修订记录和审计查询。

注意避免使用“完全自动化”“完全准确”“替代人工”等绝对表述。更合适的说法是“半自动化”“辅助著录”“降低人工整理工作量”“提高初始著录效率”。

## 9. 文档一致性基线

论文和项目文档提交前，按以下口径复查：

- 数据库权限模型以 `app_users.role` 加代码内置权限映射为准；旧 `roles`、`permissions`、`user_roles`、`role_permissions` 只应出现在“不要恢复旧 RBAC 表”或兼容命令说明中。
- 当前没有独立 `processing_job_attempts` 表；处理阶段、错误和进度追踪以 `processing_jobs` 当前状态字段和 `processing_events` 事件表为准。
- Web 上传与在线跑批已实现，但论文中应作为工程闭环能力表述，核心贡献仍放在 OCR、LLM 抽取、规则校正和可追溯著录流程。
- `DB_RERUN_POLICY` 当前只支持 `skip-success`、`rerun-failed-only`、`rerun-all`；`force-renumber` 只能作为未来高风险重排策略讨论。
- LLM 架构统一表述为外部 vLLM OpenAI 兼容接口；旧版 in-process LLM、GGUF、本地模型进程内加载不作为当前实现路径。

## 11. 推荐优先级

如果时间有限，按以下顺序完成：

1. 人工标注 30 到 50 份档案。
2. 跑出字段准确率和规则引擎消融对比。
3. 做总体架构图、流程图、规则引擎图。
4. 准备 Web 上传、进度页、人工修正截图和演示录屏。
5. 修正文档不一致。
6. 整理答辩 PPT。
